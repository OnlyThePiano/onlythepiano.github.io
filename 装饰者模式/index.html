装饰者模式

2020-06-22

一、为什么？

搭配衣服的时候，我们往往会有多种搭配方式。我们人自己是主体，而衣服是装饰，可以有根据不同情况而有多种选择。

喝奶茶的时候，我们往往也有多种加料方式。

与建筑者模式不同的是，建筑的过程是稳定的；这里来说，如果衣服的搭配是稳定的就会，搭配方式1（大T恤、大跨库）和搭配方式2（小T恤、小垮裤）等。

那么怎么才能让它们自由搭配，而不是”套餐式“搭配呢？？？？？

三、装饰者模式

装饰模式，它把每个要装饰的功能放在单独的类中，并让这个类包装它所要装饰的对象，因此，当需要执行特殊行为时，客户代码就可以在运行时根据需要有选择地、按顺序地使用装饰功能包装对象了。

“通过装饰，可以让你全副武装到牙齿，也可以让你只挂一丝到内裤。”

2）服饰类（Decorator）：

注意：protected Person component；

这里 “聚合关系” 的实现，当然也可以和责任链模式一样，在父类声明自身类型的变量给子类继承后调用父类的方法。

class Finery extends Person
{
    protected Person component;
    
    //装饰，选择想要的装饰
    public void Decorate(Person component)
    {
        this.component = component;
    }
    
    //形象展示
    public void Show()
    {
        if (component != null)
        {
            component.Show();
        }
    }

}

3）具体装饰类：

class Tshirts extends Finery
{

    public void Show()
    {
        System.out.print("TShirt");
        
        //递归输出
        super.Show();
    }
}

4）客户端代码：

//主体
Person p = new Person();
//装饰
TShirts t = new TShirts();
BigTrouser b = new BigTrouser();
Sneakers s = new Sneakers();

//开始装饰
t.Decorate(p);
b.Decorate(t);
s.Decorate(b);

//装饰结束,递归输出
s.show();

5）解释：

- 原则：

  • 开放-关闭原则的设计

- 关系：

  • 聚合关系：服饰类 聚合 主体类

  • 继承关系

通过在主体和具体装饰类之间添加一个中间层（抽象装饰类），使得具体装饰类可以通过抽象类进行通信。

子类在重写的 show() 方法里通过 super 关键字调用父类被覆盖方法，被覆盖的父类 show() 方法：先判断自己有没有包装对象，有包装的对象则调用该包装对象的 show() 方法。一步一步递归从而实现了所有具体装饰类的 show() 方法。

在重写的方法里调用被覆盖的方法，使得子类实现了又一功能。

- 递归：

  1. 各个具体装饰类包装对象，即对 Person component 赋值；
  2. 执行 show() 方法里的输出；
  3. 通过 show() 方法执行被覆盖的父类方法；
  4. 父类方法判断当前类有没有包装对象，有则调用改所包装对象的 show() 方法；
  5. 回到2步骤；

四、装饰者模式的应用

- 优点：

  • 装饰类和被装饰类可以独立发展，而不会相互耦合。它有效地把类的核心职责和装饰功能分开了

  • 装饰模式是继承关系的一个替代方案

  • 装饰模式可以动态地扩展一个实现类的功能

- 缺点：

  • 多层装饰比较复杂。比如我们现在有很多层装饰，出了问题，一层一层检查，最后发现是最里层的装饰出问题了，想想工作量都害怕

- 使用场景：　

  • 需要扩展一个类的功能时
  • 需要动态地给一个对象增加功能，并可以动态地撤销时
  • 需要为一批的兄弟类进行改装或加装功能时

- 应用实例：

  • 奶茶订单系统
