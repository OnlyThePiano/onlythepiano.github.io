泛型

所谓泛型，就是允许在定义类、接口、方法时使用类型形参，这个类型形参将在声明变量、创建对 象、调用方法时动态地指定( 即传入实际的类型参数，也可称为类型实参)。Java 5改写了集合框架中 的全部接口和类，为这些接口、类增加了泛型支持，从而可以在声明集合变量、创建集合对象时传入类 型实参，这就是在前面程序中看到的List< 和ArrayList两种类型。

java5的List接口：

一个形参类型：

public interface List<E>
{
    void add<E x>;
    Iterator<E> iterator();
}


//List<String> 等同于：
public interface ListString extends List
{
    //原来的E形参全部变成String类型实参
    void add(String x);
    Itreator<String> itreator;
}

两个形参类型：

public interface Map<K, V>
{
    //K,V可以作为类型使用
    V put(K key, V value)
    {
        
    }
}

通过这种方式，就解决虽然程序只定 义了一一个List接口，但实际使用时可以产生无数多个List 接口，只要为E传入不同的类型实参，系统就会多出一一个新的List子接口。

注意：

  • 必须指出: List 绝不会被替换成ListString， 系统没有进行源代码复制，二进制代码中没有，磁盘中没有，内存中也没有。

  • 包含泛型声明的类型可以在定义变量、创建对象时传入一个类型实参，从而可以动态 地生成无数多个逻辑.上的子类，但这种子类在物理上并不存在。

泛型的继承

  • 有传入实际类型参数时：父类的所有带有形参的地方都会被替换成实参类型；
  • 没有传入实际类型参数时：父类就默认是Object类型

这里又说一下，重写父类的原则：“两同两小一大”

  • 方法名同、形参列表同；
  • 子类返回类型比父类小或等，子类抛出异常类型比父类小或等；
  • 子类方法访问权限更小或等；

并不存在的泛型类

List<String> l1 = new ArrayList<>();
List<Integer> l2 = new ArrayList<>();
//事实上，l1、l2是相等的

不管为泛型的类型形参传入哪一种类型实参， 对于Java来说，它们依然被当成同一个类处理，在内存中也只占用–块内存空间，因此在静态方法、静态初始化块或者静态变量的声明和初始化中不允许 使用类型形参。

在静态方法、静态初始化块或者静态变量的声明和初始化中不允许使用类型形参

  • 如果Foo是Bar的一个子类型(子类或者子接口)，而G是具有泛型声明的类或接口，G并不是G的子类型；这一点非常值得注意，因为它与大部分人的习惯认为是不同的。

例如：List对象是不能当成List对象使用的。

通配符：

public void test(List<?> c){
    
}

通配符可以使得元素类型匹配任何类型，

这样任何类型的List类型来调用test都可以，因为此时List的类型时Object；而不管List的真实类型是什么，它也必定是Object类或子类。

List<?> list = new ArrayList<String>
//编译报错
list.add(new Object());

因为需要传入add方法的参数E要是E对象或其子类的对象，而此时E的类型时不知道的。java不允许把对象放入一个未知类型的集合里。

泛型方法

//声明了一个泛型方法
//带有一个类型形参
static <T> void fromArrayToCollection(T[] a, Collection<T> c)
{
    
}

这个T类型形参就可以在该方法内当成普通类型使用。

  • 与接口、类声明中定义的类型形参不同的是，方法声明中定义的形参只能在该方法里使用，而接口、类声明中定义的类型形参则可以在整个接口、类中使用；
  • 同时，不需要显式传入实际类型参数，编译器会根据实参推断；
Collection<String> cs = new Collection<>();
//当调用方法时，编译器会推断出T时String类型
fromArrayCollection(sa, cs);
