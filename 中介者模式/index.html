中介者模式

2020-06-27
Share
  Tweet  

一、为什么？

MVC 模式，C（Controller控制器）是 M（Model模型）和 V（View视图）的中介者，在前后端交互时起到了中间人的作用。

如果大量的连接使得一个对象不可能在没有其他对象的支持下工作，系统表现为一个不可分割的整体，这对系统的行为进行任何较大的改动就十分困难了。

二、中介者模式

中介者模式 (Mediator)：用一一个中介对象来封装一系列的对象交互。 中介者使各对象不需要显式地相互引用，从而使其耦合松散，而且可以独立地改变它们之间的交互。

中介者模式是基于迪米特法则。

2）具体中介者类：

class UnitedNationsSecurityCouncil extends UnitedNations
{
    private USA colleague1;
    private Traq colleague2;
    
    @Override
    private void Declare(String message, Country colleague)
    {
        if (colleague == colleague1)
            colleague2.GetMessage(message);
        else
            colleague1.GetMessage(message);
    }
}

3）具体国家类：

class USA extends Country
{
	protected UnitedNations mediator;
	
	public USA(UnitedNations mediator) {
		this.mediator = mediator;
	}
	
	public void Declare(String message) {
		mediator.Declare(message, this)
	}
	
	public void GetMessage(String message)
	{
	
	}
}

4）客户端代码：

UnitedNationsSecurityCouncil UNSC = new UnitedNationsSecurityCouncil();

USA c1 = new USA(UNSC);
Iraq c2 = new Iraq(UNSC);

//建立关联关系
//具体中介者类需要了解各个具体的国家
UNSC.Colleague1 = c1;
UNSC.Colleague2 = c2;

c1.Declare("不准研制核武器");
c2.Declare("我们没有核武器");

4）解释：

- 原则：

  • 迪米特法则

- 关系：

  • 关联关系：具体中介类关联具体国家类
  • 关联关系：抽象国家类关联抽象中介类

这一个“循环”的过程是：消息给到服务器，服务器再给到另一方。

抽象国家类中声明了抽象中介者的对象，可以调用中介者类的方法后，也就可以通过中介者和其他国家通信。

具体中介者类需要了解各个具体的国家，才能达到中介、转发的作用。

交互的复杂性变为了中介者的复杂性，直接的表现就是具体中介者关联的国家越来越多、实现国家间能通信的方法越来越复杂。

- 中介者知道每个具体国家：

  • 可以通过直接赋值给中介者：UNSC.Colleague1 = c1;
  • 可以给个函数：UNSC.addColleague(c1);
class UnitedNationsSecurityCouncil extends UnitedNations
{
    private List<Country> colleague = new ArraryList<>();
    
    public void addColleague(Country country) {
    	colleague.add(country);
    }
    
}

- 抽象类降低耦合，增加复用：

抽象类是为了更好的与客户端分离，所有存在关系的类用的都是抽象类；而实际传入的参数是具体类联合国安理会类，为了调用其抽象类定义的唯一方法 Declare()。

三、中介者模式的应用

- 优点：

  • 减少类间依赖，降低了耦合
  • 符合迪米特原则

- 缺点：

  • 中介者会膨胀的很大，而且逻辑复杂

- 使用场景：

  • 系统中对象之间存在比较复杂的引用关系
  • 想通过一个中间类来封装多个类的行为，而又不想生成太多的子类

- 应用实例：

  • 聊天系统
  • 房屋中介/机场调度系统

- 注意事项：

  • 不应当在职责混乱时使用
